# ADR-002 — Routing Protocol

## Статус
Принято — 2026-04-17.

## Контекст
В mesh-сети сообщения должны достигать получателя через промежуточные узлы.
Нужен алгоритм routing устойчивый к:
- Потере узлов mid-relay
- Циклам (loop prevention)
- Шторму трафика на больших сетях
- Приоритезации SOS над обычными сообщениями

## Варианты

### A: Flooding / Epidemic
- Каждый узел переcылает всем соседям
- **Плюсы:** простота, устойчивость, нет состояния
- **Минусы:** O(n²) трафик на больших сетях

### B: DSR (Dynamic Source Routing)
- Маршрут строится заранее
- **Минусы:** не работает в volatile сетях где узлы быстро пропадают

### C: Gossip с Bloom Filter (Bridgefy/Briar подход)
- Каждый пакет несёт bloom filter посещённых узлов
- Relay не отправляет узлу, который уже в фильтре
- **Плюсы:** O(n) в среднем, устойчиво
- **Минусы:** сложнее реализация

## Решение

**P0: Epidemic с loop prevention по `routePath: PeerId[]`**
- Простота, работает для ≤15 пиров
- `maxHops = 10`, `ttl = 24h`
- Dedup LRU по `messageId`

**P1: Переход на Gossip + Bloom Filter при обнаружении ≥15 активных пиров**
- Динамическое переключение алгоритма
- Bloom filter 256 байт, 8 хэш-функций → ~100 узлов при false positive 1%

**Priority Queue:**
- SOS (priority=3) → передаётся первым
- Обычные сообщения (priority=0-2) → FIFO
- Relay обрабатывает очередь с учётом priority

## Параметры

| Параметр | Значение | Обоснование |
|---|---|---|
| maxHops | 10 | Диаметр mesh сети в крупном здании |
| ttl | 24h | После катастрофы маршрут может восстановиться через сутки |
| dedup cache | 10 000 messageId | ~2MB памяти |
| retry per hop | 3 попытки с backoff 1s/2s/4s | Устойчивость к кратковременным потерям |
| outbox TTL | 7 дней | Для store-and-forward |

## Store-and-Forward

Если получатель недоступен:
1. Сообщение сохраняется в `mesh_outbox` (IndexedDB)
2. При появлении нового пира — проверяем outbox по пересечению `routePath`
3. Если новый пир может быть на пути — отправляем через него
4. TTL 7 дней, после — drop

## Последствия

### Плюсы
- Работает в exceptional сценариях (катастрофа)
- Простая первая версия
- Расширяется до gossip без rewrite

### Минусы
- Epidemic → трафик на больших сетях (смягчается переключением)
- Store-and-forward требует дискового пространства

## Метрики для мониторинга

- `mesh_delivery_rate` — доставлено / отправлено
- `mesh_avg_hops` — среднее число переcылок
- `mesh_dedup_hits` — сколько дубликатов отсеяно
- `mesh_outbox_size` — размер очереди pending

## Критерии приёмки
- 3 устройства линией: сообщение от A через B доходит до C за ≤5 сек
- 5 устройств клубком: 100% доставка при hop≤10
- 1000 дубликатов одного сообщения не вызывают шторм (dedup работает)
