# ADR-004 — Offline / Online Sync

## Статус
Принято — 2026-04-17.

## Контекст
Mesh-сеть работает offline, но когда у кого-то из участников появляется
интернет — мы можем использовать его как **bridge** для:
- Доставки сообщений в другую mesh-группу
- Broadcast SOS сигнала во внешний мир
- Persistence (не потерять историю при переустановке)

## Принципы

1. **Offline-first** — mesh всегда работает, сервер опционален
2. **Нет зависимости от сервера** — отказ Supabase не ломает функционал
3. **CRDT-подход** к конфликтам — все устройства могут независимо писать

## Решение

### Модель: двухсторонний опциональный sync

```
┌──────────────┐                    ┌──────────────┐
│  mesh-группа │ ←── bridge peer ──→│   Supabase   │
│  (offline)   │   (при online)     │  (онлайн)    │
└──────────────┘                    └──────────────┘
```

### Что синкается в Supabase

| Таблица | Когда | Что |
|---|---|---|
| `mesh_identities` | Первый запуск + online | Публичные ключи, prekey bundles |
| `mesh_sos_signals` | При получении SOS + online | Для широкого broadcast через push |
| `mesh_message_hashes` (опц.) | Для анти-replay бэкапа | Хэши доставленных messageId |

### Что **НЕ** синкается
- Сами сообщения (E2EE, сервер не должен их видеть)
- Приватные ключи (остаются на устройстве)

### CRDT merge правила

| Поле | Тип | Merge |
|---|---|---|
| `mesh_identities.prekey_bundle` | jsonb | LWW (last-write-wins) по `updated_at` |
| `mesh_sos_signals.resolved_at` | timestamptz | min() — кто первый разрешил, тот и прав |
| `mesh_message_hashes` | set | union — дедуп через UNIQUE |

### Race conditions

Ситуация: одно и то же сообщение доставлено и через mesh, и через Supabase.
**Решение:** дедуп по `messageId` на receiver side (который сам генерится
как `SHA-256(senderId || timestamp || nonce)` — детерминированный).

### Broadcast SOS через push

Когда bridge peer получил SOS и имеет интернет:
1. Вызывает Edge Function `crisis-mesh-sos-broadcast`
2. Функция находит пользователей в радиусе по `ll_to_earth`
3. Шлёт FCM push + email каждому
4. Фиксирует `mesh_sos_signals.broadcasted_at`

## Последствия

### Плюсы
- Расширение охвата SOS через интернет когда он есть
- Backup identities для восстановления после переустановки
- Не ломается оффлайн-режим

### Минусы
- Дополнительная логика на клиенте
- Риск утечки метаданных (кто вышел в сеть, когда)

### Privacy-by-default
- Sync выключен по умолчанию (opt-in в настройках)
- При включении — чёткое объяснение что уходит в облако
- Возможность wipe облачных данных (right to be forgotten)

## Критерии приёмки
- Offline mode работает полностью без сети
- При переключении online → offline → online: нет потерь сообщений
- Dedup работает: сообщение через mesh + через Supabase → показано один раз
- SOS broadcast через push доставляется ≤30 сек
