# ADR-001: Превращение модуля страхования в независимый агрегатор

**Статус:** proposed  
**Дата:** 2026-04-05  
**Авторы:** architect

## Контекст

Текущий модуль страхования содержит 20 страниц, 52+ компонентов, 7 Edge Functions (insurance-quote, insurance-vehicle-lookup, insurance-kbm-check, insurance-purchase, insurance-assistant, insurance-calculate, insurance-companies), 14+ таблиц БД с RLS. Однако интеграция с реальными СК работает только через mock/sandbox. Отсутствуют: CRM-кабинет агента с балансом/лояльностью, реферальная программа, полноценные черновики, пролонгации.

## Варианты

| Вариант | Плюсы | Минусы |
|---|---|---|
| A: Минимальные изменения — дополнить существующие таблицы, добавить лояльность и рефералы | ~300 строк SQL, ~800 строк TS, минимальный риск | CRM останется поверхностным, нет agent_balance view |
| B: Чистая архитектура — отдельная схема `insurance`, новая модель данных | Идеальная структура, чистые связи | ~2000 строк SQL, высокий риск миграции, дублирование данных |
| C: Прагматичный баланс — дельта-миграция к существующим таблицам + новые stores/hooks | ~600 строк SQL, ~1500 строк TS, переиспользует 80% кода | Требует аккуратных ALTER TABLE |

## Решение

Вариант C — прагматичный баланс. Существующая БД уже содержит 80% структуры (agent_profiles, insurance_clients, insurance_calculations, insurance_commissions, insurance_payouts, policy_renewals). Нужно добавить: insurance_referral_links, insurance_loyalty_history, insurance_drafts. Расширить agent_profiles полями лояльности.

## Последствия

- Положительные: переиспользование существующего кода, минимальный риск
- Отрицательные: некоторая неконсистентность имён (agent_profiles vs insurance_agents)
- Риски: ALTER TABLE на таблицах с данными

## Метрики успеха

- tsc --noEmit: 0 ошибок
- Все таблицы с RLS
- Edge Functions отвечают < 20s
- Mock-адаптер работает offline
