## 2. Как устроен калькулятор страхового агрегатора

### 2.1 UX-паттерн: Multi-step Wizard

Все крупные агрегаторы используют пошаговый wizard, а не одну большую форму.

**Sravni.ru (ОСАГО)**:
```
Шаг 1: Госномер → автозаполнение (марка, модель, мощность, VIN)
Шаг 2: Владелец (ФИО, дата рождения, паспорт)
Шаг 3: Водители (возраст, стаж, КБМ — автоматическая проверка АИС РСА)
Шаг 4: Период использования, мультидрайв
Шаг 5: Результаты → предложения от 12-18 СК, отсортированные по цене
```

**PolicyBazaar (Auto)**:
```
Шаг 1: Car number или Brand new? → автозаполнение марки/модели
Шаг 2: Детали авто (вариант, топливо, город)
Шаг 3: Результаты → 20+ planов с фильтрами
```

**Cherehapa (Travel)**:
```
Шаг 1: Куда едете? (мультивыбор стран)
Шаг 2: Даты поездки
Шаг 3: Количество туристов, возраст
Шаг 4: Результаты → отсортированные по цене предложения
```

### 2.2 Ключевые UX-паттерны

1. **Progressive disclosure** — не показывать все поля сразу
2. **Auto-fill** — госномер/VIN → марка, модель, мощность, год
3. **Мгновенный preview** — цена обновляется при каждом изменении параметра
4. **Smart defaults** — предзаполнение на основе профиля пользователя
5. **Comparison mode** — возможность добавить 2-4 предложения в сравнение
6. **Sticky CTA** — кнопка "Оформить" всегда видна при скролле
7. **Trust signals** — рейтинг компании, отзывы, claim settlement ratio

### 2.3 Формулы расчёта

**ОСАГО (регулируется ЦБ РФ)**:
```
Премия = ТБ × КТ × КБМ × КВС × КО × КМ × КС × КП × КН

ТБ  — базовый тариф (коридор ЦБ: 2746-4942₽ для легковых)
КТ  — территориальный коэффициент (1.0-2.1, по регионам)
КБМ — коэф. бонус-малус (0.46-3.92, 14 классов)
КВС — коэф. возраст-стаж (0.83-1.87)
КО  — ограничение водителей (1.0 список / 1.94 мультидрайв)
КМ  — мощность двигателя (0.6-1.6)
КС  — сезонность (0.5-1.0)
КП  — период страхования (0.2-1.0)
КН  — нарушения (1.0-1.5)
```

**КАСКО** (устанавливается СК, ~4-8% от стоимости авто):
- Базовая ставка × коэф. возраста авто × регион × франшиза × стаж × КБМ
- Сильно зависит от андеррайтинга каждой СК

**Ипотечное** (обычно 0.15-0.5% от суммы):
- Страхование конструктива + титул + жизнь заёмщика
- Срок = срок ипотеки, обязательное продление

**ВЗР/Travel** (~90-300₽/день):
- Страна × дни × возраст × покрытие × опции (спорт, беременность)

### 2.4 Pricing Engine (на примере HealthSync)

```
final_price = base_price + age_modifier + smoker_modifier + sum(selected_addons)

pricing_modifiers:
  - age 18-25: +$10 flat
  - age 26-40: +$25 flat
  - age 41-55: +$50 flat
  - age 56-65: +$85 flat
  - smoker: +20% of base
```

**Фильтрация (9 шагов, строгая последовательность):**
1. Province/region filter
2. Age eligibility filter
3. Smoking filter
4. Coverage type filter
5. Plan type filter (individual/couple/family)
6. Price calculation (apply modifiers)
7. Add-on application
8. Budget filter
9. Sort by price ASC

## 3. Как устроены API-интеграции с СК

### 3.1 Модель интеграции

Существует 3 модели подключения страховых компаний:

#### A. Прямой API (Direct Integration)
```
Наш сервер → API страховой компании → Котировка

Плюсы: Полный контроль, свежие данные, быстрый ответ
Минусы: Каждая СК = отдельная интеграция, нужен партнёрский договор
Примеры: Крупные агрегаторы (Sravni, PolicyBazaar)
```

#### B. Через агрегатор-посредника (B2B API)
```
Наш сервер → InsSmart/Cherehapa API → Множество СК → Котировки

Плюсы: Одна интеграция = десятки СК, мгновенный старт
Минусы: Зависимость от посредника, комиссия ниже, ограниченный контроль
Примеры: Начинающие агрегаторы, виджеты на сайтах
```

#### C. Виджет / iFrame (White-label)
```
Наш сайт ← iFrame генератор InsSmart/Cherehapa

Плюсы: Zero-code, мгновенный запуск
Минусы: Нет контроля UX, зависимость от провайдера, ограниченная кастомизация
```

### 3.2 Архитектура multi-provider запросов

```
┌─────────────┐
│  Calculator  │  ← калькулятор собирает данные пользователя
│   (Frontend) │
└──────┬──────┘
       │ POST /api/quote
       ▼
┌──────────────┐
│  Quote Engine │  ← Edge Function / API endpoint
│  (Backend)    │
└──────┬───────┘
       │ Promise.allSettled([...providers])
       ├──────────────────────────────────┐
       ▼                                  ▼
┌──────────────┐                  ┌──────────────┐
│ Adapter:     │                  │ Adapter:     │
│ InsSmart     │                  │ Cherehapa    │
│ (ОСАГО/КАСКО)│                  │ (Travel)     │
└──────┬───────┘                  └──────┬───────┘
       │                                  │
       ▼                                  ▼
┌──────────────┐                  ┌──────────────┐
│ СК: Ингос,   │                  │ СК: ERV,     │
│ Альфа, СОГАЗ │                  │ Абс, Ренесс │
└──────────────┘                  └──────────────┘
       │                                  │
       └──────────┬───────────────────────┘
                  ▼
          ┌───────────────┐
          │ Ranking Engine │  ← сортировка, скоринг, фильтрация
          │ + Cache Layer  │
          └───────┬───────┘
                  │
                  ▼
          ┌───────────────┐
          │  Результаты    │  ← массив предложений, отсортированных
          │  (Frontend)    │     по цене/рейтингу/покрытию
          └───────────────┘
```

### 3.3 Adapter Pattern (ключевой паттерн)

Каждый провайдер — отдельный адаптер с единым интерфейсом:

```typescript
interface InsuranceProviderAdapter {
  code: ProviderCode
  name: string
  supportedProducts: InsuranceCategory[]
  
  getQuote(request: QuoteRequest): Promise<ProviderQuote[]>
  purchase(request: PurchaseRequest): Promise<PurchaseResult>
  checkStatus(policyId: string): Promise<PolicyStatus>
  
  // Ожидаемый timeout для этого провайдера
  timeout: number  // ms
}
```

**Критические аспекты:**
- `Promise.allSettled` — НЕ `Promise.all` (одна СК упала → остальные ОК)
- Per-provider timeout (InsSmart ~5s, Cherehapa ~3s, local calc ~100ms)
- Retry с backoff для transient ошибок (network)
- Caching котировок (TTL 5-15 min для тех же параметров)
- Logging каждого запроса/ответа (для аналитики и debug)
- Fallback на локальный расчёт, если все провайдеры недоступны

### 3.4 InsSmart — модель взаимодействия

InsSmart — это **НЕ публичный API**. Это **B2B SaaS платформа для страховых агентов**:

- Регистрация как агент → получение доступа к кабинету
- Кабинет: расчёт ОСАГО/КАСКО/ипотечное/ВЗР/НС/антиклещ
- ~520K полисов/год через платформу, 10K+ агентов
- Вознаграждение: ОСАГО ~6K₽, КАСКО ~20K₽, ипотека ~30K₽
- Для B2B-интеграции: "Решение для бизнеса" — виджет/API по партнёрскому договору
- Стоимость: бесплатно для агентов (InsSmart берёт комиссию от СК)

**Вывод**: для интеграции нужно заключить **партнёрский договор** как B2B-клиент InsSmart, после чего они предоставят API или виджет.

### 3.5 Cherehapa — модель взаимодействия

- Специализация: travel-страхование (ВЗР)
- Партнёрская программа (CPA/revenue share)
- API/виджет по договору
- Для интеграции: связаться через /partners (сейчас 503)

### 6.1 Стратегия: РЕФАКТОРИНГ + УСИЛЕНИЕ, не переписывание

Текущий модуль содержит ~17K строк рабочего кода. Переписывать его с нуля — нерационально. Правильная стратегия:

1. **Сохранить** — формулы, типы, справочники, миграции, Edge Functions
2. **Переструктурировать** — UX (единый хаб вместо 20 страниц)
3. **Добавить** — реальные API, live-preview, профиль пользователя, кэшбэк
4. **Удалить** — дублирующиеся компоненты, неиспользуемые страницы

## 9. Риски и митигации

| Риск | Вероятность | Митигация |
|---|---|---|
| InsSmart/Cherehapa откажут в партнёрстве | Средняя | Локальный расчёт как fallback, другие B2B API (Ресо, Согаз прямые) |
| Слишком много mock-данных (fake aggregator) | Высокая | Начать с 1 реального провайдера (InsSmart), остальные — sandbox |
| Регуляторные риски (нужна лицензия?) | Низкая | Агрегатор ≠ страховой агент, но нужна юр. консультация |
| Перегруз Supabase (много запросов к СК) | Средняя | Кэширование, rate limiting, background jobs |
| UX разочарование (все показывают одинаковые цены) | Средняя | Добавить value: рейтинги, отзывы, кэшбэк, рекомендации |
