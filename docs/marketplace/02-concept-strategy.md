**Для партнёров (логисты, аффилиаты, разработчики):**

## 3. Полная Функциональная Спецификация

### 3.1 Система Регистрации, Верификации и Профилей

#### 3.1.1 Регистрация Покупателей

**Способы регистрации:**
- Телефон + OTP (SMS/звонок) — основной метод
- Email + пароль — альтернатива
- Google OAuth 2.0
- Яндекс OAuth
- ВКонтакте OAuth
- Telegram Login Widget (через Bot API)
- Apple Sign In (iOS only)
- Госуслуги ESIA (для B2B верификации)

**Флоу регистрации по телефону:**
```
Экран 1: Ввод номера телефона
  - Поле: +7 (___) ___-__-__  с автоформатированием
  - Кнопка "Получить код" (disabled до валидного формата)
  - Ссылка "Войти через email"
  - Чекбокс согласия с оферта + политика конфиденциальности
  - Дефолт: checked (но пользователь должен видеть явно)

Состояния кнопки:
  - Loading: spinner, disabled
  - Error: красная рамка, текст ошибки под полем
  - Rate limit: "Повторить через 59 секунд"

Экран 2: Ввод OTP-кода
  - 4 поля (или 1 поле) для 4-значного кода
  - Auto-submit при вводе 4 цифр
  - Таймер обратного отсчёта 60 секунд
  - "Отправить повторно" (активен после истечения таймера)
  - "Другой способ" → email или звонок

Business Rules:
  - OTP валиден 5 минут
  - Максимум 5 попыток → блокировка 15 минут
  - Максимум 3 SMS/10 минут с одного IP
  - Максимум 5 SMS/24 часа на один номер

Экран 3: Заполнение профиля (опционально, skip возможен)
  - Имя (обязательно для продавца, опционально для покупателя)
  - Дата рождения (для 18+ категорий, верификация)
  - Пол (необязательно, для рекомендаций)
  - Город (автоопределение по IP + ручной выбор)
```

**Edge Cases регистрации:**
- Номер уже зарегистрирован → показать "Войти?" с pre-fill номера
- Номер заблокирован антифродом → показать email-регистрацию
- Нет SMS (роуминг, виртуальный номер) → предложить звонок с кодом
- Оффлайн при вводе кода → сохранить состояние, продолжить при восстановлении

#### 3.1.2 Верификация Продавцов

**Уровни верификации продавцов:**

| Уровень | Требования | Лимиты | Преимущества |
|---------|-----------|--------|-------------|
| Basic | Телефон + email | ₽100K GMV/мес, 50 SKU | Тестовый режим |
| Standard | Паспорт РФ + ИНН + фото с документом (ФЛ/ИП) | ₽5M GMV/мес, 1000 SKU | FBO, реклама |
| Business | ИНН + ОГРН + уставные документы (ООО/АО) | Неограниченно | Бренд Store, B2B, API |
| Enterprise | KYC + compliance проверка | Неограниченно | Приоритетная поддержка, SLA 4ч |

**Процесс KYC продавца (Standard):**
```
Шаг 1: Личные данные
  - ФИО (точно как в паспорте)
  - Дата рождения
  - Серия/номер паспорта
  - Орган выдачи + дата

Шаг 2: Загрузка документов
  - Главный разворот паспорта (фото, мин. 1500×1000 px)
  - Страница с регистрацией
  - Selfie с паспортом (liveness check — модель ML определяет живое лицо)
  - ИНН (можно автопроверка через ФНС API)

Шаг 3: Банковские реквизиты
  - БИК + р/с (автопроверка через API ЦБ РФ)
  - Или привязка карты через токенизацию

Шаг 4: Договор-оферта
  - Электронная подпись (галочка + IP + timestamp)
  - Отправка копии на email

Автоматическая проверка:
  - OCR документов (собственная модель + ABBYY)
  - Проверка ФИО/ИНН в ФНС (API nalog.ru)
  - Проверка в реестре дисквалифицированных лиц
  - Санкционные списки (FinCEN, OFAC, FATF)
  - Скоринг рисков: история на других платформах (если доступно)

Сроки:
  - Автоматическое одобрение: ~40% случаев, до 15 минут
  - Ручная проверка: ~60%, до 24 часов (рабочие дни)
  - Отказ: уведомление с причиной и возможностью апелляции
```

#### 3.1.3 Профили покупателей

**Структура профиля покупателя:**
```

Приватный профиль:

  - Настройки уведомлений
```

### 3.3 Интеллектуальный Поиск

#### 3.3.1 Архитектура поиска

```
Пользователь вводит запрос
         │
         ▼
  Query Preprocessing
  ├── Spell correction (Hunspell + ML)
  ├── Lemmatization/Stemming (pymorphy3 для русского)
  ├── Query expansion (synonyms, related terms)
  ├── Intent classification (navigational / transactional / informational)
  └── Language detection (RU/EN/mixed)
         │
         ▼
  Search Engine (Elasticsearch 8.x)
  ├── BM25 full-text (title^3, description^1, tags^2)
  ├── kNN Vector Search (bi-encoder embeddings, HNSW index)
  ├── Attribute filters (facets)
  └── Boosting signals:
      ├── Relevance score
      ├── Popularity (CTR, CVR, orders last 30d)
      ├── Price competitiveness
      ├── Seller performance
      ├── Stock availability
      ├── Freshness (new listing boost)
      └── Personalization score (user history)
         │
         ▼
  Re-Ranking Layer (LambdaMART model)
  ├── Features: query-product relevance, user history, context
  └── Online learning: A/B test новых моделей
         │
         ▼
  Results Assembly
  ├── Deduplication (same product, different sellers)
  ├── Diversity injection (нет больше 3 подряд от одного продавца)
  └── Sponsored items injection (max 20% позиций)
```

#### 3.3.2 Семантический поиск

**Технология:** Bi-encoder model (ruBERT-based, fine-tuned на product search)

- Индекс: ~100 млн товарных векторов (384-dim), ~15 GB
- Метрика: cosine similarity
- HNSW параметры: M=16, efConstruction=256, ef=128
- Latency target: < 50ms для top-100 results
- Обновление индекса: near real-time (< 30 секунд после публикации товара)

**Примеры семантических запросов:**
- "подарок мужу на юбилей 40 лет любит рыбалку" → рыболовные снасти + товары для отдыха
- "что надеть на свадьбу летом" → вечерние платья, летний стиль
- "тихая мышь для офиса" → бесшумные мыши, офисная периферия

#### 3.3.3 Поиск по фото (Visual Search)

**Pipeline:**
```
1. Пользователь загружает фото (камера или галерея)
2. Image preprocessing: resize to 224×224, normalize
3. Feature extraction: EfficientNet-B5 (fine-tuned on product catalog)
4. Vector query: top-50 по cosine similarity из FAISS index
5. Re-ranking: по доступности, цене
6. Отображение: галерея "Похожие товары"
   └── Кнопка "Найти точно такой же" → поиск по barcode (если виден)
```

**Поддерживаемые сценарии:**
- Фото реального товара → аналог на NEXUS
- Скриншот из соцсетей → восстановление через метаданные + визуал
- Fashion look → поиск отдельных элементов гардероба
- Штрих-код / QR-код → прямой переход на товар

#### 3.3.4 Голосовой поиск

- **Web:** Web Speech API (Chrome, Safari)
- **Mobile:** Native speech recognition (iOS SFSpeechRecognizer, Android SpeechRecognizer)
- **Постобработка:** → текстовый запрос → стандартный поиск pipeline
- **NLU:** распознавание сложных запросов "найди мне красную женскую куртку размер 44 до трёх тысяч"

#### 3.3.5 Автодополнение

**Алгоритм:**
1. Triggered при вводе 2+ символов, debounce 150ms
2. Sources приоритетами:
   - Персональная история поиска пользователя
   - Trending queries последних 24ч
   - Popular queries по категории (если пользователь в категории)
   - Full catalog suggestions (prefix match + semantic)
3. Показывается: 5–7 подсказок максимум
4. Для каждой подсказки: текст + иконка категории + топ-3 товара с фото

**Edge Cases:**
- Опечатки: "айфон" → "iPhone" (spell-corrected suggestion)
- Транслитерация: "samsung" → "Samsung", "samsung galaxy"
- Запрос без результатов → "Похоже, у нас нет [запрос]. Может, [похожий запрос]?"

### 3.4 Система Рекомендаций

#### 3.4.1 Архитектура рекомендательной системы

**Уровни рекомендаций:**

```
Tier 1: Real-time (latency < 100ms)
├── Current session context
├── Recently viewed items
└── Cart-based recommendations

Tier 2: Near real-time (latency < 500ms)
├── Collaborative filtering (user-based, item-based)
├── Content-based filtering
└── Hybrid model output

Tier 3: Batch (обновляется каждые 4 часа)
├── Popularity-based
├── Category trends
└── Seasonal adjustments
```

**Модели:**
1. **Matrix Factorization (ALS)** — user-item взаимодействия
2. **Neural Collaborative Filtering (NCF)** — глубокое обучение
3. **Graph Neural Network (GraphSAGE на Neo4j)** — социальный граф (купили вместе, смотрели вместе)
4. **Transformer-based Sequential Model** — анализ последовательности действий

#### 3.4.2 Плейсменты рекомендаций

| Плейсмент | Алгоритм | Кол-во товаров |
|-----------|---------|---------------|
| Главная страница: "Для вас" | Hybrid NCF | 24 |
| Главная: "Тренды категории" | Category trending | 12 |
| PDP: "С этим покупают" | Association rules (Apriori) | 8 |
| PDP: "Похожие товары" | Item-based CF + content | 12 |
| PDP: "Смотрели также" | Session-based sequential | 8 |
| Корзина: "Добавьте к заказу" | Cart abandonment + associations | 5 |
| Post-purchase: "Дополните образ" | Purchase completion model | 8 |
| Email: "Вы не купили" | Abandoned cart model | 5 |
| Email: "Новинки для вас" | Interest model | 10 |
| Push: "Снижение цены" | Wishlist price alert | 3 |

#### 3.4.3 Cross-sell и Upsell

**Cross-sell правила:**
- Используется Association Rules (min_support=0.01, min_confidence=0.15)
- Обновление: ежедневно
- Логика: "Покупатели, купившие A, также купили B"
- Фильтры: не показывать товары от того же продавца если они конкурируют между собой

**Upsell правила:**
- Показать более дорогую вариацию (better specs, larger size)
- Показать товар с лучшим рейтингом в той же категории
- Показать бундл (товар + аксессуар) с суммарной скидкой 5%

#### 3.6.2 Эскроу-логика

**Принцип:** деньги покупателя удерживаются на эскроу-счёте до подтверждения получения заказа.

```
1. Покупка → средства заморожены на NEXUS Escrow Account
2. Товар доставлен + покупатель не открыл спор за 72 часа → автоматическое высвобождение
3. Покупатель подтвердил получение → немедленное высвобождение
4. Покупатель открыл спор → средства удерживаются до решения
5. Решение в пользу покупателя → возврат средств покупателю
6. Решение в пользу продавца → выплата продавцу с задержкой 24ч

Выплата продавцу:
- Стандарт: T+1 (следующий рабочий день после автовысвобождения эскроу)
- Минимальная сумма выплаты: ₽500 (или накопительно до следующей)
- Метод: банковский перевод (СБП) или на NEXUS Pay кошелёк
- Частота: ежедневно (рабочие дни)
```

**Партнёры и условия:**

| Партнёр | Период | Сумма | Одобрение |
|---------|--------|-------|-----------|
| Тинькофф Сплит | 3/6/12 мес, 0% | ₽500–₽300K | Секунды |
| Альфа Частями | 4 платежа, 0% | ₽1K–₽100K | Минуты |
| Сбер Долями | 4 платежа, 0% | ₽500–₽150K | Минуты |
| NEXUS Credit (собственный) | 6/12/24 мес | ₽5K–₽500K | 24 часа |

**Флоу BNPL:**
```
1. Пользователь выбирает "Оплатить частями"
2. Выбирает партнёра и срок
3. Redirect на страницу партнёра (или встроенный iframe)
4. Партнёр проводит скоринг (секунды-минуты)
5. Одобрено → autoredirect обратно на NEXUS с токеном
6. NEXUS подтверждает заказ
7. Партнёр выплачивает NEXUS полную сумму сразу
8. Покупатель платит частями партнёру
```

#### 3.6.4 Криптовалюты

**Интеграция:** NOWPayments API (non-custodial)  
**Поддерживаемые валюты:** USDT (TRC20, ERC20), BTC, ETH, TON, LTC

**Флоу:**
```
1. Пользователь выбирает "Оплатить криптовалютой"

4. Время ожидания платежа: 20 минут
5. QR-код + адрес кошелька
6. Подтверждение сети: 1-3 блока в зависимости от сети
7. NEXUS получает уведомление webhook от NOWPayments
8. Заказ подтверждается
9. Возврат: в NEXUS Credit (в рублях по курсу на момент возврата)
```

### 3.9 Система Отзывов и Рейтингов

#### 3.9.1 Типы отзывов

| Тип | Условия | Вес в рейтинге |
|-----|---------|---------------|
| Верифицированная покупка | Товар доставлен | 1.0 |
| Верифицированная покупка + фото | Фото прикреплено | 1.3 |
| Верифицированная покупка + видео | Видеоотзыв | 1.5 |
| Неверифицированная | Нет покупки (только для продавцов-конкурентов) | 0 (не отображается) |

**Политика:** только верифицированные покупки могут оставить отзыв. Это решает проблему накрутки на 90%.

#### 3.9.2 Блокчейн-верификация отзывов

**Технология:** Polygon (EVM-совместимый, низкие газовые расходы)

```
Процесс:
1. Заказ доставлен + покупатель пишет отзыв
2. NEXUS генерирует хэш: SHA-256(order_id + user_id + review_text + timestamp)
3. Хэш записывается в смарт-контракт Polygon
4. Транзакция: ~0,001$ в gas fees (за NEXUS)
5. Отзыв получает NFT Badge "Verified Review" с transaction hash

Польза:
- Покупатель может проверить подлинность: QR → blockchain explorer
- Невозможно удалить или изменить отзыв задним числом
- Независимый аудит без доверия к NEXUS
```

#### 3.9.3 Алгоритм борьбы с накруткой

**ML-модель (Isolation Forest + LSTM):**

**Признаки фейкового отзыва:**
- Написан в течение 5 минут после доставки (слишком быстро для реальной оценки)
- IP адрес совпадает с IP других отзывов на этот товар
- Паттерн: одни и те же 3 слова "отличный товар, рекомендую, 5 звёзд"
- Device fingerprint совпадает у нескольких разных аккаунтов
- Аккаунт зарегистрирован < 7 дней назад
- Аккаунт имеет > 20 заказов за 1 день (выкуп товаров для накрутки)
- NLP: текст позитивный, но слишком генеричный (cosine similarity с шаблонами > 0.9)

**Действия при обнаружении:**
- Confidence 90%+: автоматическое скрытие + метка "На проверке"
- Confidence 70-90%: очередь ручной проверки (модератор в течение 24ч)
- Confidence < 70%: оставить, но мониторить

### 3.13 B2B Модуль (NEXUS Business)

#### 3.13.1 Корпоративные аккаунты

**Структура B2B аккаунта:**
```
Организация (корневой аккаунт)
├── Администратор (роль: полный доступ)
├── Закупщик (роль: создание заказов до лимита)
├── Одобряющий (роль: аппрув заказов > лимита)
├── Наблюдатель (роль: только просмотр)
└── Бухгалтер (роль: финансовые отчёты)

- Разделение бюджетов по отделам

- История всех закупок с фильтрацией
- Массовый экспорт для бухгалтерии
```

#### 3.13.2 Тендерная Система

```
Тендер (RFQ — Request for Quotation):
1. Покупатель создаёт запрос:
   - Наименование / описание требуемого товара
   - Количество
   - Желаемые сроки
   - Требования к поставщику
   - Дедлайн приёма предложений

2. Система нотифицирует подходящих продавцов (matching по категориям)

3. Продавцы подают предложения (до дедлайна):
   - Цена, количество, сроки
   - Прикрепляемые документы (сертификаты, КП)
   - Возможность переговоров

4. Покупатель сравнивает → выбирает → подписывает договор (интеграция с ЭДО)

5. SLA ответа продавца: 48 часов (иначе автоматически исключается в этом тендере)
```

### 3.15 Антифрод и Безопасность

#### 3.15.1 Мониторинг транзакций

**Real-time антифрод (< 200ms решение):**

```python
# Псевдокод правил антифрода
def evaluate_transaction(transaction):
    risk_score = 0
    
    # Поведенческие паттерны
    if transaction.time_since_registration < 1_day:
        risk_score += 30
    if transaction.amount > user.average_order * 5:
        risk_score += 25
    if transaction.ip_country != user.registered_country:
        risk_score += 20
    if transaction.device_fingerprint not in user.known_devices:
        risk_score += 15
    if transaction.velocity_1h > 3:  # 3+ транзакции за час
        risk_score += 35
    
    # ML модель (XGBoost + LSTM)
    ml_score = fraud_model.predict(transaction.features)
    risk_score += ml_score * 40
    
    if risk_score > 75:
        return "DECLINE"
    elif risk_score > 50:
        return "3DS_CHALLENGE"  # Дополнительная аутентификация
    else:
        return "APPROVE"
```

#### 3.15.2 Верификация подлинности товаров

**Программа "NEXUS Authentic":**

1. **Бренд-верификация:** официальный бренд загружает серийные номера или QR-коды
2. **Проверка при поставке FBO:** случайная выборка, 5% товаров проверяется сотрудниками
3. **AI-проверка фото:** модель визуального сравнения с эталонными фото бренда
4. **Покупатель-репортинг:** кнопка "Сообщить о подделке" → SLA проверки 24ч
5. **Punishment:** продавец, у которого найден контрафакт → немедленная блокировка + hold выплат
