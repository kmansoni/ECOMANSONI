## 1. Введение и обзор

Telegram — это кроссплатформенный мессенджер с открытым исходным кодом клиентской части, разработанный Николаем и Павлом Дуровыми. Первая версия была запущена в августе 2013 года. На март 2026 года аудитория мессенджера превышает 900 миллионов активных пользователей, что делает его одним из крупнейших приложений для обмена сообщениями в мире.

### 1.1 Ключевые характеристики платформы

| Характеристика | Значение |
|--------------|---------|
| Год запуска | 2013 |
| Количество пользователей | 900+ млн |
| Аудитория СНГ | ~40% от общей базы |
| Количество языков | 60+ |
| Кроссплатформенность | iOS, Android, Windows, macOS, Linux, Web |
| Исходный код | Open Source (клиент) |
| Штаб-квартира | Дубай, ОАЭ |

Telegram позиционируется как мессенджер с акцентом на скорость, безопасность и открытость. Платформа построена на собственном протоколе MTProto и использует распределённую инфраструктуру серверов по всему миру.

## 2. Архитектура системы

### 2.1 Общая архитектура

Архитектура Telegram представляет собой многоуровневую систему, включающую клиентскую часть, промежуточный слой (MTProto Proxy), серверную инфраструктуру и дополнительные сервисы.

```
┌─────────────────────────────────────────────────────────────────┐
│                        Клиентские приложения                      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │  iOS    │  │ Android │  │ Windows │  │ macOS   │  │  Web    │  │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘  └─────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                     MTProto Proxy / Gateway                      │
│              (TCP, HTTP, WebSocket соединения)                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                       Серверная инфраструктура                   │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐    │
│  │  Frontend    │  │   Backend    │  │   Data Storage        │    │
│  │  Servers     │  │   (API)      │  │   (PostgreSQL + Redis)│    │
│  └──────────────┘  └──────────────┘  └──────────────────────┘    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐    │
│  │  Media       │  │  Auth        │  │   CDN                │    │
│  │  Storage     │  │  Servers     │  │   (Cloudinary/CDN)   │    │
│  └──────────────┘  └──────────────┘  └──────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 Компоненты серверной инфраструктуры

#### 2.2.1 Frontend Servers (MTProto Frontend)

Основная функция — обработка клиентских соединений и первичная валидация данных. Эти серверы принимают зашифрованные запросы от клиентов, расшифровывают их с использованием ключей сессии и передают данные на Backend.

- **Количество:** несколько сотен серверов по всему миру
- **Функции:** соединение с клиентами, базовая валидация, маршрутизация
- **Протокол:** MTProto поверх TCP/HTTP/WebSocket

#### 2.2.2 Backend Servers (API)

Содержит бизнес-логику приложения:

- Обработка сообщений и медиафайлов
- Управление пользователями и чатами
- Синхронизация данных между устройствами
- Обработка бот-команд
- push-уведомления

#### 2.2.3 Database Layer

- **Primary:** PostgreSQL для основных данных (пользователи, чаты, сообщения)
- **Session Storage:** Redis для кэширования сессий и горячих данных
- **Media Storage:** распределённая файловая система для медиафайлов

#### 2.2.4 CDN (Content Delivery Network)

Используется для доставки медиафайлов (изображения, видео, голосовые сообщения, стикеры) с минимальной задержкой. Telegram поддерживает несколько CDN-уровней для оптимизации доставки контента в разных регионах.

### 2.3 Архитектурные принципы

1. **Распределённость:** Сервера расположены в нескольких дата-центрах по всему миру для минимизации задержек и обеспечения отказоустойчивости.

2. **Горизонтальное масштабирование:** Каждый компонент может масштабироваться независимо. Frontend-серверы могут добавляться без изменений в Backend.

3. **Асинхронная обработка:** Сообщения обрабатываются асинхронно, что обеспечивает мгновенную доставку на уровне протокола.

4. **Минимальная задержка:** Целевое время доставки сообщения — менее 100 мс между серверами.

## 3. Сетевые протоколы

### 3.1 Поддерживаемые протоколы транспортного уровня

Telegram поддерживает несколько протоколов для установления соединения между клиентом и сервером:

#### 3.1.1 TCP (Primary)

Основной протокол для мобильных и десктопных приложений. Использует собственный бинарный протокол поверх TCP.

- **Порт:** 443 (стандартный), также поддерживаются альтернативные порты
- **Особенности:** Постоянное соединение с keepalive-пакетами
- **MTProto over TCP:** бинарная сериализация, сжатие, шифрование

#### 3.1.2 HTTP Long Polling

Используется в веб-версии и в ситуациях, когда прямое TCP-соединение недоступно (корпоративные прокси, файрволы).

- **Особенности:** Запросы с длительным ожиданием ответа (до 30 секунд)
- **Использование:** Telegram Web, некоторые мобильные клиенты при проблемах с TCP

#### 3.1.3 WebSocket

Современная альтернатива HTTP Long Polling, используемая в веб-версиях и Mini Apps.

- **Особенности:** Полнодуплексное соединение поверх одного TCP
- **Преимущества:** Меньшая задержка, меньший оверхед по сравнению с HTTP

#### 3.1.4 MTProto Proxy

Выделенный прокси-протокол для обхода блокировок и оптимизации соединения:

```
Client → MTProto Proxy → Telegram Server
```

- **Популярные публичные прокси:** t.me/proxy
- **Особенности:** Скрывает факт использования Telegram, может уменьшать задержку

### 3.2 Сравнение протоколов

| Протокол | Задержка | Надёжность | Использование |
|----------|---------|-----------|---------------|
| TCP | Низкая | Высокая | Основной |
| HTTP Long Polling | Средняя | Средняя | Web |
| WebSocket | Низкая | Высокая | Mini Apps |
| MTProto Proxy | Низкая-средняя | Высокая | Обход блокировок |

## 4. Протокол MTProto

### 4.1 Обзор MTProto

MTProto — это собственный протокол Telegram, разработанный для обеспечения высокой скорости, безопасности и надёжности. Протокол состоит из трёх уровней:

```
┌─────────────────────────────────────────┐
│           Message Layer                  │
│    (Структура сообщений, методы API)     │
├─────────────────────────────────────────┤
│           Crypto Layer                   │
│    (Шифрование, аутентификация)           │
├─────────────────────────────────────────┤
│           Transport Layer                │
│    (TCP, HTTP, WebSocket)                │
└─────────────────────────────────────────┘
```

### 4.2 Типы шифрования

#### 4.2.1 Server-Client Encryption (Cloud Chats)

Используется для обычных ("облачных") чатов. Сообщения шифруются между клиентом и сервером, но сервер имеет доступ к расшифрованным данным.

**Алгоритм:**
- **Key derivation:** PBKDF2 с SHA-256
- **Шифрование:** AES-256-IGE
- **Message authentication:** SHA-256 HMAC

```
Client                                    Server
  │                                          │
  │───── Auth Key (Diffie-Hellman) ─────────►│
  │                                          │
  │───── Encrypted Request ─────────────────►│
  │◄─── Encrypted Response ──────────────────│
  │                                          │
```

**Процесс установления ключа:**

1. Клиент генерирует пару RSA (2048 бит) или Diffie-Hellman ключей
2. Обмен ключами через RSA-OAEP
3. Создание сессионного ключа через DH
4. Все последующие сообщения шифруются симметричным ключом

#### 4.2.2 End-to-End Encryption (Secret Chats)

Используется для секретных чатов. Сообщения шифруются на устройстве отправителя и могут быть расшифрованы только на устройстве получателя. Сервер не имеет доступа к содержимому.

**Алгоритм:**
- **Key exchange:** Diffie-Hellman (2048 бит или 4096 бит)
- **Шифрование:** AES-256-IGE
- **Message authentication:** SHA-256 HMAC
- **Perfect forward secrecy:** Ротация ключей для каждого сообщения

**Процесс:**

```
Участник A                              Участник B
    │                                        │
    │────── A's public key (DH) ───────────►│
    │◄───── B's public key (DH) ─────────────│
    │                                        │
    │  [Общий секрет вычислен локально]       │
    │                                        │
    │────── E2EE Message ───────────────────►│
    │◄───── E2EE Message ─────────────────────│
```

### 4.3 Структура сообщения MTProto

Каждое сообщение в MTProto имеет следующую структуру:

```
┌─────────────────────────────────────────────────┐
│  auth_key_id (64 bits) — ID ключа аутентификации │
├─────────────────────────────────────────────────┤
│  message_id (64 bits) — уникальный ID сообщения   │
├─────────────────────────────────────────────────┤
│  seq_no (32 bits) — номер последовательности      │
├─────────────────────────────────────────────────┤
│  message_data_length (32 bits) — длина данных    │
├─────────────────────────────────────────────────┤
│  message_data (variable) — сериализованные данные │
├─────────────────────────────────────────────────┤
│  padding (variable) — дополнение до блока 16 байт │
└─────────────────────────────────────────────────┘
```

### 4.4 API методы

Telegram Bot API и Client API предоставляют сотни методов. Основные категории:

#### 4.4.1 Аутентификация

| Метод | Описание |
|-------|----------|
| `auth.sendCode` | Отправить код подтверждения |
| `auth.signIn` | Войти с кодом |
| `auth.signUp` | Зарегистрировать новый аккаунт |
| `auth.logOut` | Выйти из аккаунта |
| `auth.resetAuthorizations` | Сбросить все сессии |

#### 4.4.2 Пользователи

| Метод | Описание |
|-------|----------|
| `users.getFullUser` | Получить полную информацию о пользователе |
| `users.getUsers` | Получить список пользователей по ID |
| `account.updateProfile` | Обновить профиль |
| `account.updateStatus` | Обновить статус (онлайн/был(а) в сети) |

#### 4.4.3 Сообщения

| Метод | Описание |
|-------|----------|
| `messages.sendMessage` | Отправить сообщение |
| `messages.editMessage` | Редактировать сообщение |
| `messages.deleteMessages` | Удалить сообщения |
| `messages.forwardMessages` | Переслать сообщения |
| `messages.getHistory` | Получить историю чата |

#### 4.4.4 Чаты и каналы

| Метод | Описание |
|-------|----------|
| `messages.createChat` | Создать чат |
| `channels.createChannel` | Создать канал |
| `channels.inviteToChannel` | Пригласить в канал |
| `messages.getDialogs` | Получить список диалогов |

### 4.5 Схема авторизации

```
┌─────────────────────────────────────────────────────────────┐
│                     Схема авторизации                        │
└─────────────────────────────────────────────────────────────┘

1. Регистрация устройства:
   - Клиент генерирует auth_key
   - Отправляет phone_number
   - Сервер отправляет SMS с кодом
   - Клиент подтверждает код

2. Сессия:
   - auth_key привязан к device_id
   - Каждая сессия имеет session_id
   - Сессии могут быть отозваны удалённо

3. Токены:
   - Основной токен хранится на сервере
   - Temp token для анонимной публикации в каналах
```

### 5.2 Архитектура клиента

#### 5.2.1 TDLib (Telegram Database Library)

Основная библиотека для всех официальных клиентов (кроме iOS и Android Go):

- **Язык:** C++
- **Платформы:** Windows, macOS, Linux, Android (частично)
- **Функции:** криптография, работа с API, локальное кэширование
- **Доступ:** Open Source (github.com/tdlib)

```
┌─────────────────────────────────────────┐
│              UI Layer                    │
│    (SwiftUI, Kotlin, Qt, React)        │
├─────────────────────────────────────────┤
│            TDLib (C++)                   │
│  - API взаимодействие                    │
│  - Локальная база данных                 │
│  - Криптографические операции            │
│  - Синхронизация                         │
├─────────────────────────────────────────┤
│          System Layer                   │
│    (Файловая система, сеть, UI)         │
└─────────────────────────────────────────┘
```

#### 5.2.2 iOS (Telegram-iOS)

- **Язык:** Swift
- **UI Framework:** SwiftUI + UIKit (гибрид)
- **Особенности:** Адаптивный интерфейс для iPhone/iPad

#### 5.2.3 Android (Telegram for Android)

- **Язык:** Kotlin
- **UI Framework:** Jetpack Compose
- **Особенности:** Material Design 3, поддержка Android Auto

### 5.3 UI элементы и компоненты

#### 5.3.1 Экраны и навигация

Основные экраны приложения:

   - Закреплённые чаты

   - История сообщений
   - Поле ввода

3. **Профиль пользователя/группы**

4. **Настройки**

Telegram славится плавной анимацией:

## 7. Системы безопасности

### 7.1 Криптографическая защита

#### 7.1.1 Сравнение шифрования

| Тип чата | Шифрование | Ключ | Сервер видит |
|----------|-----------|------|--------------|
| Облачный | Server-Client | Shared | Да (метаданные) |
| Секретный | End-to-End | Только у участников | Нет |
| Звонки | SRTP | Shared | Нет |

#### 7.1.2 Криптографические алгоритмы

| Функция | Алгоритм |
|---------|----------|
| Обмен ключами | Diffie-Hellman (2048/4096-bit) |
| Симметричное | AES-256-IGE |
| Хеширование | SHA-256, SHA-512 |
| Аутентификация | HMAC |
| Key derivation | PBKDF2 |

### 7.2 Аутентификация и авторизация

#### 7.2.1 Первичная аутентификация

```
1. Регистрация:
   - phone_number → SMS код
   - code → temporary token
   - token + password (если 2FA) → auth token

2. Сессия:
   - auth_key привязывается к device_id
   - session_id генерируется для каждого входа
   - refresh_token для продления сессии
```

#### 7.2.2 Двухэтапная аутентификация (2FA)

- **Первый фактор:** SMS код
- **Второй фактор:** Пароль (устанавливается пользователем)
- **Восстановление:** Email (опционально)

### 7.3 Защита от атак

| Тип защиты | Реализация |
|------------|------------|
| MITM | Certificate pinning |
| Replay | Nonce + timestamp |
| Brute force | Rate limiting |
| XSS | Sanitization |
| SQL Injection | Parameterized queries |
| DDoS | Cloudflare integration |

### 7.4 Метаданные и приватность

Даже при шифровании сервер собирает метаданные:

| Данные | Хранение |
|--------|----------|
| Phone number | Да |
| Contacts | Да (опционально) |
| IP address | Временно |
| Время онлайн | Да |
| Тип устройства | Да |
| Взаимодействия | Да |

## 9. Синхронизация и хранение данных

### 9.1 Синхронизация между устройствами

#### 9.1.1 Модель синхронизации

Telegram использует "облачную" модель синхронизации:

```
┌─────────────────────────────────────────────────────────────┐
│                   Облачная модель                           │
│                                                             │
│  Устройство 1  ◄───────── Cloud Storage ─────────►  Устройство 2 │
│       │                                    │                │
│       │      (сообщения, медиа, контакты)   │                │
│       ▼                                    ▼                │
│  Локальный кэш                         Локальный кэш        │
└─────────────────────────────────────────────────────────────┘
```

#### 9.1.2 Что синхронизируется

| Данные | Облако | Локально |
|--------|--------|----------|
| Сообщения | Да | Кэш |
| Медиа | Да (опционально) | Опционально |
| Контакты | Да | Кэш |
| Группы/Каналы | Да | Кэш |
| Настройки | Да | Нет |
| Секретные чаты | Нет | Только на устройстве |

#### 9.1.3 Синхронизация в реальном времени

- **Протокол:** Long Polling / WebSocket
- **Задержка:** < 100 мс (обычно)
- **Механизм:**
  - Клиент подключается к серверу
  - Сервер уведомляет о новых данных
  - Клиент загружает изменения

### 9.2 Локальное хранение

#### 9.2.1 База данных клиента

| Платформа | База данных |
|-----------|-------------|
| Mobile | SQLite |
| Desktop | SQLite / Custom |
| Web | IndexedDB |

#### 9.2.2 Схема данных (упрощённо)

```
messages:
  - id (int)
  - dialog_id (int)

  - date (int)

dialogs:
  - id (int)

users:
  - id (int)
  - first_name (string)

```

### 9.3 Сессии и устройства

#### 9.3.1 Управление сессиями

- **Просмотр:** Все активные сессии доступны в настройках
- **Завершение:** Можно завершить отдельные сессии

#### 9.3.2 Multi-Device
